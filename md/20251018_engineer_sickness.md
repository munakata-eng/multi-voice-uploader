## なぜ細部にうるさいのか

プログラムは機械が読む文章です。
人間には同じに見えても、
コンピューターには別物ということが多々あります。
その代表例が「スペース」「半角/全角」
「大文字/小文字」の違いです。

## ありがちなトラブル3選

### 1. 見えないスペース問題

例: `index .html` のように、
ドット前に半角スペースが入ると
ファイル名が別物になり動きません。

- 対策
  - エディターの「不可視文字表示」をON
  - ファイル名はコピー&ペーストで厳密に

### 2. 半角/全角の取り違え

日本語環境特有の落とし穴です。
`１２３`(全角)は数値ではなく文字列として扱われ、
計算に使えません。全角スペースも要注意です。

- 対策
  - 入力モードを都度確認（英数/かな）
  - Linter/Formatter を導入して自動修正

### 3. 大文字/小文字の不一致

`Image.png` と `image.png` は、
環境によっては別ファイルとして扱われます。
ローカルで動いても本番で落ちる典型です。

- 対策
  - すべて小文字・ハイフン区切りに統一
  - Git でのリネームは慎重に実施

## 初心者向けチェックリスト

- 余分なスペースが入っていないか
- 数字・英字は半角か（全角混入なし）
- ファイル名は小文字+ハイフンで統一
- エディターの不可視文字表示はONか
- 保存前に Linter/Formatter を走らせたか

## すぐ試せる実践例

- VS Code での設定
  - 設定検索で「render whitespace」をON
  - 「Format on Save」をON
  - 拡張機能: ESLint/Prettier や
    文字種チェック系を導入

- 文字列→数値の変換確認
  - JavaScript: `Number("１２３") // NaN`
  - Python: `int("１２３")  # エラー`

## セキュリティと品質の観点

- パス名の乱れは参照ミスを誘発し、
  バンドル漏れ・公開漏れの原因に。
- 検証環境と本番環境の差異（OS差）で、
  思わぬ障害につながることも。
- 自動チェック（CI）を用意すると、
  人為ミスを早期に検知できます。

## まとめ

- スペース/半角全角/大文字小文字は、
  動作を左右する重要ポイント。
- エディター設定と自動整形で
  事故を未然に防げます。
- 困ったらまず「見えない差異」を疑う、
  これがエンジニアの基本動作です。
